🌸 Цветочный календарь Беларуси 🌺

Бизнес-логика проекта

Нам нужно написать поисковик цветения в Беларуси. То есть, сайт, который показывает, в каком городе и когда цветет определенный цветок в Беларуси. Сайт простой, одностраничный, выводит на экран по запросу пользователя цветы, цветущие в регионе в заданном диапазоне времени.

Сам сайт выглядит таким образом:

1.  Сверху в левом углу логотип в виде цветочка.

2.  По центру название сайта - "Найди свой цветок".

3.  Далее, под названием сайта, расположен переключатель в меню: 2 кнопки, одна из них Города, а другая Растения. По умолчанию при заходе на сайт активна кнопка Города и режим Города: это значит, что поиск будет происходить среди городов Беларуси. Если пользователь выбирает кнопку Растения, то сайт переключается в режим Растения, и поиск будет происходить по растениям.

Пример: пользователь в режиме "Города" ввел город Минск, указал интересующие даты, и нажал кнопку Поиск, то в результате поиска ему отобразится на странице ниже полей ввода список растений, цветуших в данный момент в Минске, а также их иллюстрации и описание. Если пользователь выбрал кнопку "Растения", указал интересующие даты, и нажал кнопку Поиск, то в результате поиска ему отобразится на странице список городов, где цветет это растение в интересующие даты.

1.  сразу после этого переключателя с двумя кнопками расположена основная рабочая область сайта: поисковик растений, цветущих сейчас либо же в любую другую дату. Он выглядит как 2 поля с возможностью ввода и кнопка Найти цветы. Первое поле служит для ввода названия города Беларуси, либо, если выбран режим Растения, ввода названия растения. Это поле имеет автозаполнение: когда пользователь начинает вводить интересующий его город или растение, система предлагает ему подсказку, и если пользователь нажмет на нее, то это поле становится автоматически заполненным. Второе поле служит для выбора диапазона дат. При нажатии на него чуть ниже поля ввода открывается мини-календарь с датами в формате месяц, и пользователь может кликнуть на интересующую его дату в этом либо любом другом месяце, после чего это поле будет автоматически заполнено. Далее пользователь нажимает на кнопку Найти цветы, и происходит поиск цветущих растений в выбранном городе, после чего они отображаются внизу страницы, сразу ниже основной рабочей области сайта. Если же выбран режим Растения, то при нажатии на кнопку Найти цветы происходит поиск городов, в которых цветет выбранное растение, и они отображаются внизу страницы, сразу ниже основной рабочей области сайта.

2.  После основной рабочей области сайта с посиком цветов, ниже, располагается блок с 3 карточками под названием Сейчас цветет+выбранный город (по умолчанию Минск, если пользователь выбирает другой город, то 3 карточки, что цветет в этом городе). Этот блок показывает, какие 3 любых цветка цветут в данном городе прямо сейчас. По клику на карточку пользователь попадает на страничку с описанием этого растения и в какой местности природы его можно найти, либо же в каких заказниках, либо же в каких парках и садах.

3.  Теперь про те растения либо города, которые будут отображаться внизу страницы в результате поиска на сайте - хотелось бы, чтобы они были в виде интерактивной карточки с иллюстрацией растения/города и коротким описанием из 2 предложений. По клику на эту карточку открывается новая страничка, где пользователю подробно описывается этот цветок, и в какой местности природы его можно найти, либо же в каких заказниках, либо же в каких парках и садах. Если это город, то должно быть описание достопримечательностей города с главными парками и описанием, какие растения можно найти в этих парках.

1.  По клику на логотип сайта происходит возврат на стартовую страницу сайта либо же очищаются все поля ввода - они становятся снова пустыми.

Техническая документация проекта
================================

«Цветочный календарь Беларуси»
==============================

1) Что надо сделать
-------------------

Нужно разработать веб-приложение, которое помогает пользователю узнать:

-   какие растения цветут в выбранном городе Беларуси в выбранный период;

-   в каких городах Беларуси цветет выбранное растение в выбранный период;

-   что цветет в выбранном городе прямо сейчас;

-   подробную информацию о растениях и городах на отдельных страницах.

По сути, это поисковая система по календарю цветения, построенная вокруг двух сущностей:

-   Города

-   Растения

И двух сценариев поиска:

-   Поиск по городу

-   Поиск по растению

Продукт должен быть:

-   одностраничным в основной части;

-   быстрым;

-   визуально приятным;

-   понятным обычному пользователю;

-   легко расширяемым по данным;

-   пригодным для добавления новых городов, растений, сезонов, карточек и статей.

* * * * *

2) Как это должно работать
--------------------------

2.1. Общая логика сайта
-----------------------

После открытия сайта пользователь попадает на главную страницу.\
На ней есть:

-   логотип;

-   заголовок сайта;

-   переключатель режима поиска;

-   поле поиска;

-   поле выбора диапазона дат;

-   кнопка запуска поиска;

-   блок "Сейчас цветет в [город]";

-   блок с результатами поиска.

Сайт работает в двух режимах.

### Режим 1. Города

Пользователь выбирает режим Города.\
Тогда первое поле поиска означает: введите город.

Сценарий:

1.  Пользователь начинает вводить "Мин..."

2.  Сайт предлагает подсказки: "Минск", "Минская область" и т.д.

3.  Пользователь выбирает "Минск".

4.  Пользователь выбирает диапазон дат.

5.  Нажимает кнопку "Найти цветы".

6.  Сайт показывает список растений, которые цветут в Минске в этот период.

### Режим 2. Растения

Пользователь выбирает режим Растения.\
Тогда первое поле поиска означает: введите растение.

Сценарий:

1.  Пользователь начинает вводить "сак..."

2.  Сайт предлагает подсказки: "Сакура" и т.п.

3.  Пользователь выбирает нужное растение.

4.  Пользователь выбирает диапазон дат.

5.  Нажимает кнопку "Найти цветы".

6.  Сайт показывает список городов, где это растение цветет в указанный период.

* * * * *

2.2. Что такое "диапазон дат"
-----------------------------

Сайт ищет по диапазону дат. Это означает:

-   дата начала;

-   дата окончания.

Пример:

-   10 апреля --- 25 апреля

Тогда система должна искать все растения, чье цветение пересекается с этим диапазоном.

Формально логика такая:

Если у растения период цветения:

-   с 5 апреля по 20 апреля,

а пользователь выбрал:

-   с 10 апреля по 15 апреля,

то растение подходит.

Если пользователь выбрал:

-   с 21 апреля по 30 апреля,

то растение уже не подходит.

### Правило пересечения диапазонов

Объект подходит, если:

floweringStart <= selectedEnd\
и\
floweringEnd >= selectedStart

Это ключевая бизнес-логика поиска.

* * * * *

2.3. Блок "Сейчас цветет"
-------------------------

Под поиском показывается блок:

"Сейчас цветет в Минске"

По умолчанию:

-   выбран город Минск, если пользователь еще ничего не вводил.

Если пользователь затем выбирает другой город, например Гродно, блок должен меняться на:

"Сейчас цветет в Гродно"

В этом блоке показываются 3 карточки растений, которые цветут на текущую дату.

### Важное правило

Этот блок не должен зависеть от поля диапазона дат.\
Он всегда показывает актуальное цветение на сегодня для выбранного или дефолтного города.

* * * * *

2.4. Что происходит по клику на карточку
----------------------------------------

Есть два типа карточек:

### Карточка растения

По клику открывается отдельная страница растения, где есть:

-   название;

-   фото;

-   короткое описание;

-   период цветения;

-   города Беларуси, где встречается;

-   природные зоны;

-   парки, ботанические сады, заказники, леса, аллеи, где можно увидеть;

-   советы по наблюдению;

-   возможно, карта или список мест.

### Карточка города

По клику открывается отдельная страница города, где есть:

-   название города;

-   краткое описание;

-   основные парки, сады, зеленые зоны;

-   какие растения там можно увидеть;

-   сезонные особенности;

-   подборка цветущих растений по месяцам.

* * * * *

2.5. Поведение логотипа
-----------------------

По клику на логотип сайт должен:

-   либо возвращать на главную страницу;

-   либо, если пользователь уже на главной, очищать поиск и сбрасывать интерфейс в дефолтное состояние.

Лучше сделать так:

### Если пользователь на внутренней странице

Переход на главную.

### Если пользователь уже на главной

Сброс:

-   режим = Города;

-   поле поиска пустое;

-   даты пустые;

-   дефолтный город для блока "Сейчас цветет" = Минск;

-   результаты поиска очищены.

Это предсказуемое и удобное поведение.

4.3. Главная таблица связей "растение-город-период"
---------------------------------------------------

Вот это ядро всей системы.

Потому что одно и то же растение:

-   в одном городе может цвести раньше,

-   в другом позже.

Значит, нельзя хранить просто "сакура цветет 10--20 апреля" глобально.\
Нужно хранить цветение в контексте конкретного города.

Нужна таблица вроде:

### flowering_records

-   id

-   cityId

-   plantId

-   startDate

-   endDate

Пример:

-   Минск + сакура + 12 апреля -- 24 апреля

-   Брест + сакура + 5 апреля -- 18 апреля

-   Витебск + сакура + 20 апреля -- 2 мая

Это и есть основа поиска.

5.1. Пользовательский результат

Пользователь открывает сайт и сразу понимает:

8) Как мы поймем, что все работает как надо
-------------------------------------------

Нужны четкие критерии приемки.

8.1. Функциональные критерии
----------------------------

### Главная

-   сайт открывается без ошибок;

-   активен дефолтный режим;

-   логотип работает;

-   переключатель режимов работает;

-   автокомплит работает;

-   календарь работает;

-   поиск выдает корректные данные.

### Поиск по городу

Если выбран город и период, показываются только те растения, чьи периоды пересекаются с выбранными датами.

### Поиск по растению

Если выбрано растение и период, показываются только те города, где растение цветет в эти даты.

### Блок "Сейчас цветет"

Показывает только актуальные на сегодня записи для выбранного города.

### Детальная страница

Открывается по клику и содержит корректные данные.

----------------

-   пользователь понимает, что делать без инструкции;

9) Как ничего не сломать в процессе
-----------------------------------

Вот тут уже чисто инженерная дисциплина.

9.1. Сразу разделить проект на слои
-----------------------------------

Нужно разделить:

-   UI-компоненты

-   бизнес-логику

-   data access layer

-   типы

-   utils

-   страницы

Нельзя смешивать всё в одном месте.

* * * * *

9.2. Зафиксировать типы данных
------------------------------

Надо строго описать:

-   City

-   Plant

-   FloweringRecord

-   SearchParams

-   SearchResult

И не передавать по приложению "что попало".

TypeScript здесь обязателен.

* * * * *

9.3. Не писать поиск прямо в компонентах
----------------------------------------

Плохой путь:

-   сделать фильтрацию прямо внутри React-компонента.

Правильный путь:

-   вынести бизнес-логику в отдельный сервис.

Например:

-   searchFlowersByCity(params)

-   searchCitiesByPlant(params)

-   getCurrentlyBlooming(cityId, currentDate)

Тогда UI можно менять, не ломая поиск.

* * * * *

9.4. Использовать миграции базы
-------------------------------

Нельзя менять таблицы руками на проде без истории.

Нужно:

-   Prisma migrations;

-   сиды;

-   dev/stage/prod окружения.

9.5. Делать фичи поэтапно
-------------------------

Правильный порядок:

1.  сначала данные;

2.  потом поиск;

3.  потом главная;

4.  потом карточки;

5.  потом detail pages;

6.  потом улучшения.

Не наоборот.

9.6. Писать тесты на критическую логику

-   пересечение дат;

-   поиск по городу;

-   поиск по растению;

-   выбор "сейчас цветет";

--------------------------

Нужен seed, который всегда поднимает проект в рабочем состоянии.

Например:

-   Минск

-   Гродно

-   Брест

-   Сакура

-   Магнолия

-   Яблоня

-   Одуванчик

---------------------------------

Перед продом:

-   выкатывать на staging;

-   проверять вручную;

-   только потом в production.

* * * * *

9.9. Не смешивать "реальные данные" и "UI-заглушки"

Очень частая ошибка:

-   в начале разработчик вставляет моковые данные,

-   потом начинает поверх них строить логику,

-   потом сложно переключиться на реальные данные.

Лучше сразу сделать слой адаптера:

-   либо данные из seed,

-   либо данные из базы,

-   но через единый интерфейс.

Для каждой задачи фиксировать:

-   написан код;

-   работает локально;

-   есть типы;

-   нет линтер-ошибок;

-   есть базовые тесты;

-   проверен mobile;

-   проверены empty states;

-   задача задокументирована.

Итоговая формулировка проекта
=============================

Цветочный календарь Беларуси --- это поисковый веб-сервис, который позволяет пользователю находить цветущие растения по городам Беларуси и датам, а также узнавать, где и когда можно увидеть конкретное растение.\
Продукт строится на структурированной базе данных цветения, имеет простой двухрежимный поиск, карточки результатов и детальные страницы растений и городов.\
Архитектурно проект лучше реализовать на Next.js + TypeScript + PostgreSQL + Prisma с акцентом на качественные данные, чистую бизнес-логику и аккуратный адаптивный интерфейс.
